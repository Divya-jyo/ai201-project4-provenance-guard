# Planning: Provenance Guard

## 1. Detection signals

**Signal 1 -- LLM judgment (Groq, `llama-3.3-70b-versatile`).**
Sends the submitted text to the model with a prompt asking it to judge,
holistically, whether the text reads as human-written or AI-generated.
Captures semantic and stylistic coherence -- the kind of "this reads off"
intuition that's hard to reduce to a formula. Output: `{"ai_likelihood": float 0-1, "reasoning": str}`.
Blind spot: black box. We don't know exactly what cues it leans on, it can
be confidently wrong, and it inherits whatever bias the base model has about
what "AI writing" looks like.

**Signal 2 -- Stylometric heuristics (pure Python, no network).**
Computes sentence-length variance and type-token ratio (vocabulary diversity)
and combines them: `ai_likelihood = 0.5*(1 - norm_variance) + 0.5*(1 - norm_ttr)`.
Captures a structural tendency: AI text is often more uniform in sentence
length and word choice; human writing is more irregular. Output:
`{"ai_likelihood": float 0-1, "sentence_length_variance": float, "type_token_ratio": float}`.
Blind spot: this is a population tendency, not a law, and it needs enough
text to be meaningful. Calibration testing (see below) showed it's
essentially noise under ~100 words -- TTR is inflated for short samples
because there's no room for word repetition yet.

**Combining them:** `combined_score = 0.75*llm_score + 0.25*stylometric_score`.
LLM weighted higher because calibration testing showed the stylometric
signal is unreliable on short text; it still contributes a quarter of the
score so it can pull the result when it strongly disagrees with the LLM.

## 2. Uncertainty representation

A combined score is a 0-1 "AI-likelihood." We deliberately do NOT split
at 0.5. Because a false positive (telling a human their work was flagged
as AI) is worse than a false negative on a creative platform, the
threshold for "likely_ai" is set high and the threshold for "likely_human"
is set lower -- it should be easier to land in "uncertain" than to land in
a confident accusation.

- `combined_score >= 0.70` -> `likely_ai`
- `combined_score <= 0.30` -> `likely_human`
- otherwise -> `uncertain`

The user-facing `confidence` is **not** the raw combined score. It's the
distance from the 0.5 midpoint, rescaled to 0-1:
`confidence = abs(combined_score - 0.5) * 2`. A combined score of 0.95 maps
to confidence 0.90 (very confident AI); a combined score of 0.55 maps to
confidence 0.10 (barely confident at all) -- which is exactly the case the
spec calls out: 0.51 confidence should read very differently from 0.95.

## 3. Transparency label design

Exact text per variant (see README for the canonical copy + live examples):

- **High-confidence AI:** "This content has been flagged as likely AI-generated by our automated detection system (confidence: {confidence_pct}%). This is an automated assessment and may be incorrect. If you wrote this yourself, you can appeal this classification."
- **High-confidence human:** "Our system found no strong signals of AI generation in this content (confidence: {confidence_pct}%). This reflects an automated assessment, not a verified guarantee of authorship."
- **Uncertain:** "Our system could not confidently determine whether this content is AI-generated or human-written (confidence: {confidence_pct}%). Treat this classification as inconclusive."

## 4. Appeals workflow

Any creator can submit an appeal via `POST /appeal` with `content_id` and
`creator_reasoning`. The system: (1) looks up the original submission by
`content_id`, (2) sets `status = "under_review"`, (3) stores the reasoning
and an appeal timestamp on that same audit-log row, (4) returns a
confirmation message. No automated re-classification. A human reviewer
opening the appeal queue (i.e. querying `GET /log` or the DB directly)
would see the original text, both signal scores, the original label, and
the creator's reasoning side by side.

## 5. Anticipated edge cases

- **Short submissions (under ~100 words).** The stylometric signal is
  unreliable here (see calibration note above) -- variance and TTR don't
  have enough room to be meaningful, so the combined score leans almost
  entirely on the LLM signal. Confirmed directly: all four reference test
  inputs (3-4 sentences each) scored within 0.06 of each other on the
  structural signal alone, despite being intuitively very different.
- **Plain, repetitive human writing** (children's writing, minimalist
  style, certain poetic forms that lean on repetition) will look AI-like
  to the stylometric signal specifically, because low sentence-length
  variance and repeated vocabulary are exactly what it associates with AI
  text. This is the asymmetric-risk scenario the threshold design exists
  to soften: an "uncertain" classification protects this writer, where a
  binary system would mislabel them confidently.

## Architecture

```
SUBMISSION FLOW
  POST /submit (text, creator_id)
        |
        v
  [stylometric_signal(text)] --ai_likelihood--+
        |                                      |
  [llm_signal(text)] --ai_likelihood-----------+--> [scoring.combine_scores]
                                                          |
                                                          v
                                                 [scoring.classify] -> attribution
                                                          |
                                                          v
                                            [scoring.generate_label] -> label text
                                                          |
                                                          v
                                              [db.insert_submission] (audit log)
                                                          |
                                                          v
                                   response: {content_id, attribution, confidence, label}

APPEAL FLOW
  POST /appeal (content_id, creator_reasoning)
        |
        v
  [db.get_submission(content_id)] -- 404 if missing
        |
        v
  [db.file_appeal] -> status="under_review", appeal_reasoning logged on same row
        |
        v
  response: {content_id, status: "under_review", message}
```

A submission flows through both signals independently, gets combined into
one score, classified against the asymmetric thresholds, turned into label
text, and logged -- all before the response goes back. An appeal doesn't
re-run any of that; it just attaches a creator's reasoning to the existing
audit-log row and flips its status, leaving the original decision intact
for a human reviewer to compare against.
